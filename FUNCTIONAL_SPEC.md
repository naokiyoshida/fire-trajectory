# 機能仕様書: fire-trajectory

- プロジェクト名: fire-trajectory (自由への軌道)
- バージョン: 2.0
- 最終更新日: 2026/05/10
- 役割: FIRE 計画の意思決定材料となる数値を月次で自動更新するシステム

---

## 1. 全体アーキテクチャ

```
[Windows タスクスケジューラ] ─起動→ [scripts/run-sync.ps1] ─実行→ [Node.js / Playwright]
                                                                       │
                                                                       ├─ マネフォME ログイン (storageState 再利用)
                                                                       ├─ 取引履歴スクレイプ (/cf?year=Y&month=M)
                                                                       ├─ 資産スナップショット (/bs/portfolio + /bs/liability)
                                                                       └─ Google Sheets API 書き込み
                                                                       │
                                                                       ▼
                                                                [Google Sheets]
                                                                       │
                                                                       ▼
                                                                [GAS スクリプト]
                                                                  setupSettings / setupReports
                                                                       │
                                                                       ▼
                                                                [Claude プロジェクト]
                                                                  Drive コネクタ経由で参照
```

数値層 (本リポジトリ) とナラティブ層 (Claude プロジェクト「資産運用計画」) の責任分担:

- 数値層: 実データの収集と数式ベースの予測。客観値の管理
- ナラティブ層: 目標年齢、戦略、前提条件の議論。主観・判断の管理

## 2. 階層分離アーキテクチャ

スクレイピング部分は5層に分離されており、マネフォME のサイト変更に対する影響範囲を局所化できる。

| 層 | ファイル | 責務 | サイト変更時の影響 |
|---|---|---|---|
| Navigator | `navigator.ts` | URL遷移、月送り、ヘッダー検証 | ボタン位置の変化 |
| Extractor | `extractor.ts` | DOM → 生データ | クラス名・構造の変化 |
| Selectors | `selectors.yml` | CSS セレクタ集（フォールバック付き） | **大半の変化はここだけ** |
| Schema | `schema.ts` | zod による型・整合性検証 | フィールド増減 |
| Transformer | `transformer.ts` | 内部表現 → Sheets 形式 | データフォーマット変化 |

各層は独立してテスト可能で、`tests/scrapers/**` に純関数のユニットテストがある（35本、vitest）。

> **シート命名規則**: 数値層のシート名はすべて日本語に統一されている (`取引履歴` / `資産推移` / `手動入力資産` / `設定` / `月次収支` / `カテゴリ別支出` / `純資産推移` / `資産配分`)。Sheets API でアクセスする際は `'シート名'!範囲` 形式の引用が必要。Node 側は `app/core/sheets-client.ts` の `quoteSheetName` ヘルパーが、GAS 側は `quoteSheetName_` 関数が引用を担う。FIRE 月次予測（旧「シミュレーション」シート）は engine.ts ＋ `dist/fire.html` に移行済みで Sheets には持たない（§4）。

## 3. データモデル

### 3.1 「取引履歴」シート

Node が `Google Sheets API` で書き込み。

| 列 | 名称 | 型 | 説明 |
|---|---|---|---|
| A | ID | string (SHA256) | 重複排除用ハッシュ。`SHA256(date-content-amount-source#occurrence)`。**category は意図的に除外**（マネフォME はカテゴリを後から編集でき、含めると編集のたび別 ID 化して二重追記されるため）。自然キー `date-content-amount-source` が衝突する取引には出現順 occurrence(0,1,2…) を付与し、同日・同額・同口座・同内容の別取引（例: 同日に ¥10,000 ATM 出金を2回）も別 ID として保持する。occurrence は「スクレイプ時の MF 返却順」から毎 run 決定的に同値が得られるため二重追記も防ぐ。ハッシュ方式変更時の移行は「シート値から ID 再計算」（旧 `remap-ids`）が**廃止**された（過去フル同期が金額を `¥` 付きで保存しており現行スクレイパ出力と一致せず破綻するため）。代わりに「再 `npm run sync` → `npm run dedupe-rows`」で旧 ID 行を素性ベースに除去する（DEPLOY_GUIDE §8.6）。**残余エッジ**: MF が同一自然キー取引の相対並び順を run 間で入れ替えた場合のみ、当該取引が一度だけ再追記され得る（無言の取りこぼしより検知容易） |
| B | 日付 | string (`YYYY/MM/DD`) | 取引日 |
| C | 内容 | string | 取引内容 |
| D | 金額 | string (Sheets が数値推論) | 円。マイナス = 支出 |
| E | 保有金融機関 | string | 例: 「エポスカード (直樹)」 |
| F | 大項目/中項目 | string | 例: 「食費/食料品」 |
| G | 取得日時 | string (ISO 8601) | スクリプト実行タイムスタンプ |

**重複排除**: 既存A列のIDセットを取得して、新規分のみ追記。
**振替・除外**: マネフォME 上で「振替」または「計算対象外」フラグの行はスキップ。

### 3.2 「資産推移」シート（資産スナップショット）

Node が月1回 1行追記。当月分が既存ならスキップ（冪等）。表示は日本語ヘッダーだが、Node 側は `ASSETS_KEY_ORDER` という内部キー定義を保持し、列順を一意に決定する。

| 列 | 表示ヘッダー | 内部キー | 型 | 説明 |
|---|---|---|---|---|
| A | 基準日 | snapshot_date | string (`YYYY-MM-DD`) | スナップショット日（JST） |
| B | 預金・現金 | cash | int | 預金・現金・暗号資産（外貨建て口座も含む） |
| C | 株式（現物） | stocks_listed | int | 株式（現物） |
| D | 株式（未上場） | stocks_unlisted | int | 「手動入力資産」の `未上場株式` から |
| E | 投資信託 | funds | int | 投資信託 |
| F | 年金 | pension | int | 年金（DC含む） |
| G | ポイント | points | int | ポイント・マイル |
| H | その他資産 | other_assets | int | 上記カテゴリに該当しない資産（マッピング外を集約） |
| I | 資産総額 | total_assets | int | scraped 合計 + stocks_unlisted |
| J | クレジット未払 | credit_card | int | クレジットカード未払残高 |
| K | 住宅ローン | mortgage | int | 住宅ローン残高 |
| L | その他負債 | other_loans | int | その他ローン（自動車、奨学金、その他の負債） |
| M | 負債総額 | total_liabilities | int | 負債総額（マネフォME 表示値） |
| N | 純資産 | net_worth | int | 資産総額 − 負債総額 |
| O | 備考 | notes | string | 「手動入力資産」の `備考` から |

**整合性検証** (zod refine):
- 資産: `cash + stocks_listed + funds + pension + points + other_assets ≈ total_assets_mf`（誤差±1円）
- 負債: `credit_card + mortgage + other_loans ≈ total_liabilities_mf`

### 3.3 「手動入力資産」シート

ユーザーが管理。マネフォME に未連携の資産・補足メモ用。

| 列 | 名称 |
|---|---|
| A | 項目 |
| B | 値 |
| C | 備考 |

「項目」列で受け付けるラベルは厳密に以下の2種:
- `未上場株式` → 内部キー `stocks_unlisted` (int 円。「資産推移」の D 列に書き込まれる)
- `備考` → 内部キー `notes` (string。スナップショットに添える自由テキスト)

ラベルが一致しない行は無視される。GAS は本シートに雛形（`未上場株式`/`備考` 行）と
右側 E:H 列の説明パネルを自動生成する。

**未上場株式（インテグレ等）をここに入れるべきか**（重要・二重計上注意）:
未上場株式は MF のポートフォリオ自動取得（`/bs/portfolio`）には**含まれない**。
`ScrapedAssetsSchema` に `stocks_unlisted` は無く、`buildAssetSnapshot` は
`total_assets = scraped.total_assets_mf + manual.stocks_unlisted` として合算する
（`app/scrapers/assets/transformer.ts`）。
- **原則**: 本シートの `未上場株式` 行に評価額を入力する（MF に登録していても
  スクレイプ対象の資産総額には反映されないため）。
- **例外**: MF 側で未上場株式が `/bs/portfolio` の「資産総額」に既に含まれて
  表示されている場合は、本シートに入れると**二重計上**になるため空欄(0)にする。
- 判定: `/bs/portfolio` の「資産総額」にその評価額が入っているかで決める。

### 3.6 集計シート群

GAS の `setupReports()` が QUERY 関数ベースで構築。データはすべて「取引履歴」「資産推移」を参照。

## 4. シミュレーション計算ロジック

### 4.1 実質利回り（フィッシャー方程式）

数式は「設定」シートを **項目名で `INDEX/MATCH` 参照**する（`dashLookup_`、行番号非依存）。

- 年間実質利回り `r = ((1 + n) / (1 + i)) - 1`
  - `n`: 名目利回り（設定: 運用利回り_名目）
  - `i`: インフレ率（設定: インフレ率）
- 月次実質利回り `r_m = (1 + r)^(1/12) - 1`
- 資産更新式: `P_t = (P_{t-1} + CF_t) × (1 + r_m)`
- **すべて「今日の円（実質）」で計算**。給与・基本生活費はインフレ連動＝実質ほぼ
  一定とみなし割戻さない。一方、年金・退職一時金は ねんきん定期便等の「将来名目額」
  で入力されるため、経過 t（月）に対し `÷ (1 + i)^(t/12)` で実質に割り戻して加算する
  （割り戻さないと数十年後の購買力を過大評価し FIRE 時期が楽観側へずれるため）。

### 4.1a 分配金課税（NISA 差別化・外国源泉）

本モデルは**総リターン1本**（`運用利回り_名目` に値上がり＋分配を合算で入れる）。
分配金を生活費に充てるか再投資するかは期末資産に影響しない（分配の受取＝部分売却と
等価）。ただし**課税は口座種別で非対称**：

| 口座 | 国内税(20.315%) | 外国源泉税(米株≒10%) |
|---|---|---|
| 特定（課税） | 課税 | 外国税額控除で概ね回収 → 実質 国内率のみ |
| NISA | 非課税 | **回収不能**（控除すべき国内税が無い）→ 残る |

値上がり益は売却まで非課税繰延なのでドラッグ対象外。これを次式の**税ドラッグ**で表す。

- 税ドラッグ（年・名目）
  `d = 分配金利回り × [ (1 − NISA比率) × 0.20315 + NISA比率 × 外国源泉割合 × 0.10 ]`
  - 第1項＝特定口座分の国内課税、第2項＝NISA 口座分の外国源泉税（NISA でも残る分）
  - `分配金利回り` は**総資産に対する**分配の割合（例: 本ポートフォリオで VIG・EDV・
    高配当ファンド計 ≒324千円/年 ÷ 総資産29.25M ≒ 1.1%）
  - `NISA比率` は**分配金のうち** NISA で非課税受取になる割合（資産割合ではない。
    分配はリスク資産に偏在し DC 年金は分配ゼロのため、分配額ベースで効かせる）
  - `外国源泉割合` は**分配金のうち**外国源泉（米株配当等）の割合（本ポートフォリオは
    VIG・SBI・V米国高配当が主で ≒0.8。国内源泉は SBI日本高配当のみ）
- 実効名目利回り `n' = 運用利回り_名目 − d` を §4.1 の `n` の代わりに用いる
- `分配金利回り = 0`、または `NISA比率 = 1 かつ 外国源泉割合 = 0` で `d = 0`
  ＝従来の総リターン1本と一致（後方互換）

本ポートフォリオの既定（分配1.1%・NISA100%・外国8割）では国内ドラッグ0・外国ドラッグ
`0.011 × 1 × 0.8 × 0.10 ≒ 0.088%/年`。小さいが**恒久的**で、NISA に米株高配当を置く戦略の
数少ない実コスト（特定口座なら外国税額控除で消せる分）。`n'` から導く `r_m` を青（期末
資産）・橙（I列）・前進シミュすべてに一本で効かせるため、線間の整合（§4.3）は保たれる。
実値は MF 保有明細・口座種別から設定する。なお DC 年金（確定拠出）の出口課税（退職所得／
雑所得）は別軸のため本モデルは未計上。

### 4.1b 年金の繰上げ/繰下げ・物価スライド

公的年金は**受給開始年齢で受給額が変わる**。`本人年金_年額`/`配偶者年金_年額` は
「**65歳で受け取る標準額**（ねんきん定期便の65歳見込額）」で入力し、開始年齢に応じた
倍率 `pensionFactor` で実受給額へ換算する（年金開始年齢スライダーと連動）。

| 開始年齢 | 倍率 | 制度 |
|---|---|---|
| 60歳（繰上げ） | 0.76 | 0.4%/月 減（▲24%） |
| 65歳（標準） | 1.00 | 増減なし |
| 70歳（繰下げ） | 1.42 | 0.7%/月 増（+42%） |
| 75歳（繰下げ） | 1.84 | +84%（上限） |

受給開始は 60〜75 歳にクランプ。これが無いと「開始年齢を遅らせるほど受給が減る」という
**制度と逆の誤示唆**になる（旧仕様の不具合）。資産が枯渇しない範囲では繰下げが有利
（増額 > 取り崩し）、枯渇する家計では繰下げ期間の取り崩しが勝って不利、と正しく振る舞う。

さらに年金は**名目額固定だとインフレで実質目減り**し、35年で実質ほぼ半減して FIRE 評価を
過度に悲観へ倒す。日本の年金はマクロ経済スライドで部分的に物価連動するため、**物価スライド率**
`pensionIndexation`（0..1）で表す:

`年金実質(t) = 65歳標準額 × pensionFactor(開始年齢) ÷ 12 × deflator^(pensionIndexation − 1)`

- `0`: 名目固定（インフレ分すべて実質目減り＝従来 `÷deflator`・**後方互換**）
- `1`: 実質固定（購買力一定）
- 既定 `0.5`: マクロ経済スライド相当の部分連動（名目固定は悲観すぎ・実質固定は楽観すぎの中間）

`pensionFactor`・`pensionIndexation` とも engine（唯一の正）で計算し、青（期末資産）・
橙（I列）・前進シミュに同一系列で効かせる（§4.3 の線間整合を保つ）。`0` で従来モデルと一致。

### 4.2 ライフイベント制御

「設定」シートの値に基づき、月次キャッシュフローを動的に切り替える。
収入は **家計入金モデル**（各自の手取りのうち家計に入れる額のみ。ボーナスは年額を月割で平準化）。

- **収入**:
  - 本人: リタイア予定日まで「本人月収_家計入金 + 本人ボーナス_年額_家計入金/12」
  - 配偶者: 配偶者退職予定日まで「配偶者月収_家計入金 + 配偶者ボーナス_年額_家計入金/12」
    （退職時期は `配偶者退職予定日`、または UI の `配偶者退職年齢` スライダーで上書き＝本人 `selfRetireAge` と対称）
  - 本人年金: 本人年金開始年齢（デフォルト65）到達月以降（将来名目額入力→実質割戻し）
  - 配偶者年金: 配偶者年金開始年齢（デフォルト65）到達月以降。本人と独立に繰上/繰下を設定できる（同上）
  - 本人退職時一時金: 本人リタイア予定月に加算（将来名目額入力→実質割戻し）
  - 配偶者退職時一時金: 配偶者退職予定月に加算（デフォルト 0 = 当てにしない、同上）
- **支出**:
  - 基本生活費: 全期間
  - 住宅ローン: ローン完済予定日まで
  - 息子支援: 息子支援終了日まで
  - 退職後社会保険料: リタイア予定日以降（就労時は手取り入金額に内包のため計上せず、
    退職で社会保険控除が消える分を国保・介護保険料として加算。実額は通知後に更新）

### 4.3 FIRE必要資産ライン（I列）

各月について「就労収入が途絶えても年金だけで `FIRE必要資産_目標年齢` まで
資産が `FIRE必要資産_目標残額` を下回らない最低期首資産」を末尾から逆算:

`I(r) = nextReq/(1 + r_m) − 年金(r) + 支出(r)`
（`nextReq` は通常 `I(r+1)`、目標年齢の最終月では `FIRE必要資産_目標残額`）

- 年金・支出とも**実質（今日の円）で単位を揃える**。インフレ前提は E列
  （就労プランの期末資産トラジェクトリ）と統一する：**支出は実質固定**
  （`÷deflator` しない＝生活費は購買力一定）、**年金のみ名目固定**で
  `÷ (1+i)^(t/12)` 実質割戻し。これにより青（期末資産）と橙（I列）が同一の
  漸化式・同一の傾き挙動になり整合する。〔以前は I列だけ支出も `÷deflator`
  していたため、青＝実質固定／橙＝名目固定と前提が逆転し、年金開始後の
  傾きの振る舞いが両線で食い違い、高インフレで橙が逆傾きになる等の不整合が
  生じていた。これを撤去。〕
- **自己整合（重要）**: I列は「**その月に退職したら**」を問う指標。よって
  `支出(r)` は就労プランの E列ではなく**退職済み前提の支出系列**を使う
  ＝退職後社会保険料は `リタイア予定日` ではなく**当月 r 以降ずっと**計上
  （ローン/息子支援は各終了日依存のまま）。これを怠ると r が
  `リタイア予定日` より前の月で I列が過少になり、就労収入で底上げされた
  期末資産が誤った低いラインを「超えて」見える（旧仕様の不具合）。
- 退職一時金は「働き続けて初めて得られる」ため **当てにしない**（保守側）。
- **FIRE可能時期** = 「**その月に退職した場合**、年金のみ・退職済み支出・
  一時金なしで前進シミュした終了年齢資産が `fireTargetRemain` 以上になる
  最初の月」。I列の意味（到達＝その月に辞めれば成立）と厳密に一致し、初接触で
  確定する。就労プランの E列がこの後 I列を割り込んでも（働き続けて
  `リタイア予定日` に辞めた場合の軌道の話なので）「その月に辞める」判断とは
  無関係で fireDate は揺らがない。「以降ずっと割り込まない」を要求していた
  旧ルールは I列自身の意味と矛盾し「到達しても未達」を生んだため廃止。
- **verdict（盤石/余裕/達成/未達）** は「fireDate に退職した場合の終了年齢
  資産（fireEndAssets：年金のみ・退職済み支出・一時金なし）」で判定。fireDate
  と**同一の前進シミュ**で算出するので両者は必ず整合する（fireDate があれば
  未達 にはならない）。就労プランの終了資産は参考値として別途保持・表示
  （負＝早期退職が有利のサイン）。
- この自己整合化により engine はレガシー GAS シミュとは**設計上意図的に
  乖離**していた。engine が唯一の正であり、レガシー GAS シミュ／FIRE射程／
  パリティ機構は撤去済み（§4.5）。

### 4.4 シミュレーションエンジン抽出とインタラクティブ HTML

**目的**: 設定の調整と結果確認をシート往復なしの1画面で行う。スライダーで
設定を動かすと「期末資産 vs FIRE必要資産」グラフと FIRE 可能時期が即時に
追従する。計算ロジックは**純粋関数 `engine.ts`（唯一の正）**に一本化し、
表示はローカル生成 HTML（`file://`、完全プライベート・認証/ホスト不要）。

**構成**:
- `app/sim/engine.ts`: `simulate(params: SimParams): SimResult` の**唯一の正**。
  §4.1〜4.3 の数式（フィッシャー実質利回り・ライフイベント CF・I列逆算）を
  TS 純粋関数で実装。`import` ゼロ（Date/Math のみ）で完全自己完結。I列
  算術は `app/gas/contract.ts` の `fireNeedValue` が独立再実装し engine.test
  の同一性アンカーになる。
- `app/sim/load-inputs.ts`: 設定シートを項目名→値で読み（読み取り専用）、
  `現在の資産`（資産推移 最新の数式結果）も含めて `SimParams` を組む。
- `app/sim/render-html.ts`: engine.ts のソースを **`typescript` の
  `ts.transpileModule` で単一ファイル JS 化**（新規依存なし・バンドラ不要・
  ロジック二重化なし＝ドリフト不能）し、入力 `PARAMS` とスライダー定義
  `SLIDERS` と共に `dist/fire.html` へインライン。グラフは依存を増やさない
  ため**手描き Canvas**（2本線＋Y/X 軸目盛りと値・Y は 0起点で期末資産に
  合わせてスケールし可視域 [0, top] の外（序盤の巨大な FIRE必要資産＝上端
  超／枯渇後の負の期末資産＝0未満）は描かず途切れさせる（偽の水平線を
  出さない・枯渇は赤い縦ラベルで明示）・X は描画幅に応じ数年おきに間引き・
  マウスオーバーで全系列の生値・FIRE可能/枯渇の縦ラベル・DPI 対応・
  リサイズ追従）。スライダー
  操作の再計算は全期間ブラウザ内 JS で完結（サーバ往復なし）。
- **分配金リアリティチェック**: ヘッドライン直下に「分配金（年額・概算グロス）と
  基本生活費に対するカバー率」を現在／退職時の2点で表示。`dividendYield × 資産`の
  概算で、「分配で生活費」戦略の妥当性を可視化する（残りは年金＋資産取り崩しで充当）。
  退職時アンカーは `SimResult.assetsAtRetire / retireYm`（エンジンが返す＝UI 側で
  退職月を再計算せずドリフトを防ぐ）。`分配金利回り=0` のときは非表示。
- **ライフイベント可視化**: グラフに退職（本人）・FIRE可能時期・資産枯渇の縦
  ラベルを表示（退職月はエンジンが返す `retireYm` を使い UI で再計算しない）。
  ツールチップは本人・配偶者の年齢を併記。未達時はヘッドラインに枯渇年齢
  （本人年齢つき）と改善レバー（生活費・退職年齢・年金開始・想定利回り）の
  ヒントを出す。
- **想定寿命の一本化**: `fireTargetAge`（想定寿命）スライダーが I列の地平・FIRE
  判定・グラフ終了年齢（`simEndAge`）を同値に揃え、二重地平の不整合をなくす
  （recompute で `simEndAge = fireTargetAge`）。
- CLI: `npm run sim`（読み取り専用で `dist/fire.html` 生成、`dist/` は
  gitignore）。`npm run sync` 末尾でも best-effort 生成（失敗しても sync は
  落とさない）。

**フル精度＋プロファイル**: engine は常に**全 ~30 設定項目**で計算する。
UI に出すスライダーは宣言的配列 `SLIDERS`（`key/label/min/max/step/unit/
profile`）で定義し、`profile` で `detailed`(PC) と `simple`(将来スマホ) を
出し分ける。露出していない項目は設定シートの値を既定として使う。**スライダーの
追加・範囲変更・プロファイル変更は `SLIDERS` の1行編集のみ**（engine/UI の
コード変更不要）。初期 PC プロファイルは高感度の約14項目
（利回り・インフレ・年金のインフレ連動・基本生活費・ローン/息子支援・退職後社保・
本人月収/ボーナス・各リタイア年・想定寿命・各年金開始年齢・各年金額・配偶者退職年齢）。

### 4.6 保守性・デバッグ容易性・AI コスト最適化（設計原則）

本機能は「修正・デバッグのたびに AI を多投する」事態を構造的に避ける:

- **単一の正**: 数式は engine.ts の1箇所のみ。Sheets と HTML でロジックを
  二重化しない（remap-ids 事故＝二重定義のドリフトを繰り返さない）。
- **自己診断**: `npm run doctor`（シート診断）と `npm test`（engine の
  不変条件を固定）、`dist/fire.html` のデバッグ表（実質係数・年金実質列を
  含む）で、異常はユーザーが1コマンド実行→要約貼付で足り、AI の探索
  フェーズを省略できる。
- **宣言的設定**: スライダー/プロファイル/設定マッピングはデータ駆動。
  多くの「変更要望」はコード変更でなく config 1行で完結（ユーザー自身でも可）。
- **決定的・オフライン・認証不要**: 再現性が高く、ホスティング/認証の
  不確実性に起因するデバッグ往復が発生しない。
- **シナリオ命名ユニットテスト**: ライフイベント規則ごとに名前付きテスト。
  失敗時に壊れた規則がテスト名で即特定でき、ローカルで回帰を閉じる。
- **HTML 内デバッグ出力**: 画面のトグルで月次表を貼り付け可能な形で
  ダンプ。不具合時はユーザーが表を貼るだけで原因箇所を共有できる。

## 6. 設計原則

### MUST
- 既存の取引・資産データを破壊しない（重複排除と冪等性で担保。さらに追記前ガードが「増分なのに過半数新規」を検知して二重追記前に停止し、`npm run doctor` で保存ID vs 現行方式の整合を随時検証できる）
- 失敗時にユーザーに気づける形で通知（数ヶ月空白を防ぐ）
- 取得項目の小計と合計の整合性を検証（zod refine）

### SHOULD
- セレクタは外部 YAML 化し、コード変更なしで追従可能にする
- スクレイピング対象は月1回に絞り、過剰アクセスを避ける
- ログを残す（成功・失敗・件数）

### MUST NOT
- マネフォME に **Google OAuth でログインしない**（Playwright 検出で弾かれる）
- 認証情報をリポジトリにコミットしない
- 過去のスナップショットを上書きしない
- 住信SBIネット銀行のような「意図的に未連携」の口座を勝手に追加しない
